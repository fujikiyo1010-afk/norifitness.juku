## 2. データベース設計 (20 テーブル)

**メインソース**: `database_design_draft.md` (2026-05-19、16 → 20 テーブルに増加)

### コアテーブル (8)
- **users** ― 受講生マスタ (email, name, status: active/lifetime/withdrawn/refunding, joined_at, support_until)
- **user_profiles** ― 受講生プロフィール (birthday, address のみ、height/gender は削除)
- **courses** ― コース (5 大カテゴリ)
- **chapters** ― 章
- **lessons** ― レッスン (vimeo_url, released_at で段階公開)
- **lesson_progress** ― 学習進捗 (is_completed で「できた!」ボタン、watched_seconds)
- **comments** ― コメント (公開・スレッド型、parent_comment_id で返信対応)
- **goal_sheets / goal_sheet_revisions** ― 目標管理シート (jsonb 柔軟構造、編集履歴追跡)

### 試験 (2)
- **tests** ― 試験 (多肢選択、passing_score)
- **test_attempts** ― 受験履歴 (何度でも受験可、unique 制約なし)

### 管理・運用 (4)
- **admin_users** ― 管理者 (superadmin / admin ロールのみ、食事スタッフなし)
- **invitations** ― 招待リンク (token, expires_at: 48 時間)
- **stripe_events** ― Stripe Webhook 記録
- **broadcast_notifications** ― お知らせ配信

### 補助 (2)
- **notifications** ― 受通知
- **user_status_history** ― ステータス変更履歴
- **faq_categories / faqs** ― FAQ

### 将来統合 (1)
- **graduate_voices** ― 卒業生事例 (01_tokuten/voices-data.json スキーマ踏襲)

**重要設計判断**:
- RLS (Row Level Security) で受講生は自分データのみ、管理者は全閲覧可
- 食事関連テーブルはなし (trainercloud で完結)
- trainercloud との連携機能なし (OEM 型 SaaS のため永久に別アプリ)

## 6-B. 月次添削動画の完全フロー (Phase 2-7 合意)

### 撮影 → 保存 → 配信 → 視聴 までの全 15 ステップ

| # | 主体 | 場所 | アクション | データの状態 |
|---|---|---|---|---|
| 1 | 受講生 | 受講生アプリ (`monthly_review_form.html`) | 17 項目を記入 → プレビュー → 送信 | `monthly_audits` テーブルに INSERT (submitted_at にタイムスタンプ) |
| 2 | のり氏 | 管理画面 (`admin_monthly_inbox.html`) | 未返答リストに新着表示、受講生を選択 | DB 読み込みのみ |
| 3 | のり氏 | 管理画面 (`admin_monthly_review.html`) | 受講生の 17 項目回答を確認 | DB 読み込みのみ |
| 4 | のり氏 | 管理画面 (`admin_monthly_record.html`) | 動画録画モード起動 | カメラ・マイク権限取得 |
| 5 | のり氏 | 同上 (準備中) | カメラ ON 確認、左で 17 項目テキスト見ながら準備 | ブラウザがカメラ映像をライブ表示 (まだ録画していない) |
| 6 | のり氏 | 同上 (録画中) | [録画開始] → 5 分以内で話す | ブラウザの MediaRecorder API がメモリに動画 Blob 蓄積 |
| 7 | のり氏 | 同上 (プレビュー) | [録画停止] → 自分で再生確認 | 動画は **ブラウザメモリの Blob URL** で再生可能、まだサーバーには行っていない |
| 8 | のり氏 | 同上 | [送信] ボタン 1 クリック | 送信処理開始 |
| 9 | システム | 裏処理 (Vimeo へアップロード) | Vimeo Upload API で動画 Blob を Vimeo に POST | **動画ファイル本体が Vimeo クラウドに保存**、進捗バーで表示 (5 分動画なら 30 秒〜2 分) |
| 10 | Vimeo | Vimeo クラウド | トランスコード (録画 → 各解像度の MP4 / HLS に変換) | Vimeo 側で自動処理、数十秒〜数分 |
| 11 | Vimeo | Vimeo クラウド | 動画 ID + 埋め込み URL を返す | 例: `https://vimeo.com/123456789` |
| 12 | システム | 裏処理 (Supabase 更新) | `monthly_audits.nori_video_vimeo_url` と `nori_video_published_at` を UPDATE | **DB には URL の文字列のみ**、動画ファイルは入らない |
| 13 | システム | 通知配信 | プッシュ通知 + LINE Messaging API で「のりfitness から返信が届きました」配信 | 強制送信 (受講生側で OFF 不可) |
| 14 | 受講生 | アプリ (`monthly_review_history.html`) | 通知に気付いてアプリ起動 → 履歴ページの当月カードが「D 返信届いた」状態に | DB 読み込みで status 判定 |
| 15 | 受講生 | アプリ (`monthly_review_detail.html`) | 月詳細画面で Vimeo Player 埋め込み再生 | Vimeo Player が **Vimeo クラウドから直接動画ストリーミング** (Supabase は経由しない) |

### データの保存先まとめ

| データ | 保存先 | 永続性 |
|---|---|---|
| **動画ファイル本体** (mp4 / hls) | **Vimeo クラウド** | 永久 (削除しない方針) |
| **動画 URL の文字列** | **Supabase DB** の `monthly_audits.nori_video_vimeo_url` | 永久 |
| **17 項目テキスト + 受講生の回答** | **Supabase DB** の `monthly_audits.items` (jsonb) | 永久 |
| **動画長さ / 解像度メタ** | **Vimeo 側** で管理 (API で取得可能) | 永久 |

### 受講生 / のり氏それぞれの過去動画閲覧

| 主体 | 経路 | 範囲 |
|---|---|---|
| **受講生 (在籍中)** | `monthly_review_history.html` → 月別カード → `monthly_review_detail.html` | **自分の全過去動画** (Vimeo URL は永久保存) |
| **受講生 (退会後)** | アクセス権切れる (RLS で制御) | 視聴不可 (データは残る、再入会時に復活) |
| **受講生 (再入会)** | RLS 戻す | **過去全動画が閲覧可能になる** |
| **のり氏 / 管理者** | 管理画面 (`admin_monthly_inbox.html` で受講生検索 → 過去履歴閲覧) | **全受講生の全過去動画** |

### Vimeo と Supabase の役割分担

```
[Vimeo の役割]
- 動画ファイル本体の保存
- トランスコード (各解像度・各形式への変換)
- iOS Safari / Android Chrome 等 全ブラウザ対応
- 再生品質保証 (CDN 経由配信)
- 再生分析データ

[Supabase の役割]
- 動画 URL の文字列保存 (DB)
- 「どの月次添削にどの動画が紐づくか」の管理
- 受講生 / 管理者のアクセス権制御 (RLS)
- 動画ファイル本体は保存しない (Storage 圧迫なし)
```

### コスト影響

- **Vimeo**: 既存 Business プラン契約内で当面運用可能。受講生 48 人 × 月 1 件 × 5 分 = 月 240 分の新規アップロード、年間 2,880 分 (約 48 時間)。Business プランの容量制限を要確認
- **Supabase**: 動画分の Storage 圧迫なし。Pro $25/月で受講生 500 人くらいまで余裕 (Tech Check 3 修正版)

### 技術検証参照
- 録画方式: ブラウザ標準 **MediaRecorder API** (Tech Check 2 で検証済、`/tmp/tech_check_mediarecorder.html`)
- 受講生視点プレビュー: 同 Tech Check 2 で「同じ Blob を 2 つの video 要素に渡す」仕組みを検証済
- **Vimeo API 連携検証** (アップロード自動化) は Phase 3 着手時に Tech Check 4 として追加検証予定

## 17. Phase 2-7 補足: 体組成記録 MVP 外化の影響

2026-05-25 きよむさん決定: 体組成記録の独立画面 (body_metrics.html) は MVP 外、目標シート ① 現状把握への手動入力欄に統合。これに伴う他機能への波及:

### 影響 1: 体脂肪率計算ツールの連携先変更

| 項目 | 元方針 | 新方針 |
|---|---|---|
| 計算結果の反映先 | 目標シート ① + body_metrics テーブル の両方 | **目標シート ① のみ** (jsonb 内の `current_status.body_fat_pct` を更新) |
| UI への影響 | tools_body_fat.html に「体組成記録に反映」チェックボックスがあった | **チェックボックスを削除** (反映先は目標シート ① 固定) |

### 影響 2: カーボサイクル「今日の目安」機能 → 削除確定 (C 案、2026-05-25 きよむさん決定)

カーボサイクル機能は「曜日ごとに糖質量を上下させる」設計で、元の体組成記録ページ上部に「今日 = HIGH 日 / LOW 日、糖質目安 X g」を表示する計画だった。体組成記録ページ MVP 外化に伴い、表示場所がなくなった。

**最終決定**: **C 案 = 「今日の目安」機能自体を削除、PFC・カーボサイクル ツール内の週次表のみ保持**

- ✅ 残すもの: `tools_pfc_carb.html` 内の週次糖質配分表 (月〜日 の HIGH/MID/LOW + 各日の糖質 g 数)
- ❌ 削除するもの: 「今日は ◯曜日、糖質目安 X g」のホーム画面 / 目標シート / その他での日次表示
- 受講生は週次表をツール画面で確認する運用 (毎日のホーム表示はしない)

**理由**: シンプル優先、ホーム画面の詰め込み回避、Phase 2-7 言語化フェーズで「過剰演出を抑制」の原則と整合

### 影響 3: 身長 (height_cm) の保存場所確定 (2026-05-26 追加)

Phase 0 で `user_profiles` から `height_cm` を削除した経緯あり (trainercloud 完結方針 + 新サイトには身体情報を最小限に)。一方、Phase 2-7 モック (goal_sheet_v3.html) では「身長 プロフィール連動 readOnly」と表記しており、DB 設計と矛盾していた。

**最終決定 (2026-05-26 きよむさん)**: **goal_sheets.content.current_status.height_cm に手入力で保存**

- ✅ 採用: 目標シート ① 現状把握で身長を手入力 (必須)
- ❌ 不採用: user_profiles に height_cm カラムを追加する案 (DB マイグレーション + 設計変更を避ける)
- 身長は普段変わらない情報のため、目標シート ① で 1 回入力すれば実用上問題なし
