## 🔍 新規選定が必要な技術（3つ）

### 1. フロントエンドフレームワーク

#### 比較表

| FW | 言語 | レンダリング | 学習コスト | Claude/Cursor との相性 | エコシステム | おすすめ度 |
|---|---|---|---|---|---|---|
| **Next.js 15** | React/TS | SSR/SSG/RSC | 中〜高 | ⭐⭐⭐ 標準 | ⭐⭐⭐ 巨大 | ⭐⭐⭐ |
| **Astro** | 多言語OK | SSG 中心 | 中 | ⭐⭐ | ⭐⭐ | ⭐⭐ |
| **Remix** | React/TS | SSR 中心 | 中〜高 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| **SvelteKit** | Svelte/TS | SSR/SSG | 低〜中 | ⭐⭐ | ⭐ | ⭐ |
| **Nuxt 3** | Vue/TS | SSR/SSG | 中 | ⭐⭐ | ⭐⭐ | ⭐ |

#### 私の推奨: **Next.js 15**

**理由**:
1. **Claude / Cursor との相性が圧倒的に最強**
   - 補完精度が他のFWより高い
   - 既存コード参考も豊富
2. **React は学習リソースが最も豊富**（公式ドキュメント + 日本語記事 + YouTube）
3. **Supabase 公式サポート**（Auth Helpers）が手厚い
4. **Vercel との統合が最速**（同じ開発元）
5. **Cloudflare Pages でもデプロイ可能**（既存利用との連携可）

**懸念**:
- App Router（Next.js 13+）の概念がやや複雑（RSC = React Server Components）
- → ただし、Cursor で補完してもらえばクリア可能

#### 代替案
- **Astro**: コンテンツ中心サイトなら最速だが、動的機能が多い本サイトには Next.js が筋
- **Remix**: 良いFWだが、エコシステムが Next.js より小さい

### 2. ホスティング

#### 比較表

| サービス | 料金 | 特徴 | のりfitness での既存利用 | おすすめ度 |
|---|---|---|---|---|
| **Cloudflare Pages** | 無料〜 | 既存利用あり（01_tokuten, 02_kinniku_lp 等）、超高速、料金安い | ✅ あり | ⭐⭐⭐ |
| **Vercel** | 無料〜（Pro $20/月）| Next.js 作成元、最高の DX | ❌ なし | ⭐⭐⭐ |
| **Netlify** | 無料〜 | 老舗、機能豊富 | ❌ なし | ⭐⭐ |

#### 私の推奨: **Cloudflare Pages**

**理由**:
1. **既存ノウハウ活用**（01_tokuten 等で既に利用、デプロイ知見あり）
2. **料金が圧倒的に安い**（無料枠が大きい、有料も Vercel より安い）
3. **超高速**（Cloudflare の CDN 網）
4. **Next.js も問題なくデプロイ可能**（@cloudflare/next-on-pages 経由）
5. **データ転送量無制限**（Vercel は有料プランでも上限あり）

**懸念**:
- Vercel ほど Next.js への最適化はしていない（ただし実用上は問題なし）
- Edge Functions 周りで若干癖がある（Supabase Edge Functions と棲み分け）

#### 代替案
- **Vercel**: Next.js の最高 DX が欲しいなら、月 $20 払って Vercel
  - 受講生数100名超で本格運用するなら Vercel 推奨

|---|---|---|---|
| **Resend** | 無料 3,000通/月〜（$20/月 50,000通）| Next.js との相性最高、React Email でリッチメール | ⭐⭐⭐ |

#### 私の推奨: **Resend**

**理由**:
1. **Next.js + React Email との相性最強**（テンプレを JSX で書ける）
2. **無料枠が広い**（月3,000通 = 受講生100名規模なら十分）
3. **設定が簡単**（API キー1つで開始）
4. **配信ログ・分析機能**が充実

## 🔧 Stripe Webhook 設計の叩き台

決済外部完結方式での Webhook 受信フロー:

```
[受講生] Stripe 決済リンクで支払い完了
   ↓
[Stripe] Webhook イベント送信
   - イベント: `checkout.session.completed` または `invoice.paid`
   - 送信先: 新サイトの Webhook エンドポイント
   ↓
[新サイト] Supabase Edge Function で受信
   - 署名検証（Stripe Webhook Secret）
   - 顧客情報を取得（メアド・名前・決済額）
   - DB に「決済済み・招待待ち」レコード作成
   ↓
[新サイト] 管理者に通知（Slack or メール）
   - 「○○さんが決済完了しました、招待してください」
   ↓
[管理者] 新サイト管理画面で「招待」ボタン押下
   - 招待メール送信（Supabase Auth）
   ↓
[受講生] メールのリンクから登録 → ログイン
```

### MVP では「**Webhook 受信 → 管理者通知 → 手動招待**」が最低限
- 完全自動 ID 発行（Webhook → 自動招待メール）は Should Have

### 必要な Webhook イベント
- `checkout.session.completed`: 一括支払い完了時
- `payment_intent.payment_failed`: 決済失敗時（管理者通知）
- `charge.refunded`: 返金処理時（10日返金保証）

- 🔴 **5週間の実装期間はかなりタイト**
- 🟡 デザイン2週間も厳しい（多画面、スマホファースト）

### 短縮策
1. **MVP 機能をさらに削る**（例: 試験機能を Should Have へ）
2. **シンプル UI**（凝らない、Tailwind UI 等のテンプレ活用）
3. **AI による開発加速**（Cursor + Claude フル活用）
4. **早期テスト**（5名規模のクローズドβ）

## ⚠️ リスクと対策

| リスク | 対策 |
|---|---|
| **8月末公開の工数オーバー** | MVP 機能の段階リリース（最初は M-1〜M-5 だけ公開、徐々に追加） |
| **1人開発の限界** | AI コードレビュー（Claude / CodeRabbit）活用、Cursor で開発速度UP |
| **テスト不足** | クローズドβ（5〜10名）で実運用前に検証 |
| **動画移行コスト** | Vimeo URL 引き継ぎだけなので低リスク |
| **既存会員の移行失敗** | 移行は完成後、許可者のみ。失敗時は旧エキスパで対応継続 |
| **Stripe Webhook の不達** | 冗長化（手動招待ボタンも残す）、リトライ機構 |
| **Supabase の Rate Limit** | Pro プラン（$25/月）で余裕、必要に応じて Read Replica |
