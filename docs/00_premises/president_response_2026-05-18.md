## 🚨 最重要：前提を大きく変える4点

### 1. 新サイトの KPI は「売上」ではなく「**生徒の実施完工率**」
> 新機能のKPIは売上ではなく、生徒の実施完工率です。
> 生徒が使いやすいサイトか、生徒の進捗を管理しやすいかが重要になってくる。
> 売上には少し関与するのですが、間接的な効果はあると思われるが重要指数ではない。
> 入会者の満足度や結果にコミットすることで、評判がよくなる。
> セールスするときに、サイトがあるから正しく学べることを訴求することで売上向上に期待できる。
> 売り上げ向上はマーケティングでの課題なので、今回は関係ないです。

**反映:**
- 主軸: 生徒の実施完工率
- 副軸: 売上向上（間接効果として、きよむさん意向）
- 売上回復は別軸（マーケ施策）で動いている

### 2. **新サイトでは決済しない**
> 新サイトでは決済はしない。

**反映:**
- 決済は Stripe 決済リンク等で**新サイト外**で完結
- 新サイトは Stripe Webhook を受けて **受講生 ID 自動発行**
- 申込フォーム・決済 UI・返金処理は**新サイトに不要**
- 大幅な工数削減・セキュリティリスク低減

### 3. trainercloud は **テスト段階（β版）**
> ⇨完了している（=コードは完成）
> （きよむさん補足: テスト段階、運用はまだまだ、修正中）

**反映:**
- "完了" の正確な意味は「コードベースは存在する」程度
- 実運用はまだ整っていない、β版
- **MVP では疎結合**を維持、フェーズ3 で SSO 連携検討

### 4. 面談成約率の更新
- 80%目標 → **95%成約**（実績ベース）

## 💸 決済外部化の具体フロー（提案）

```
[受講生が個別面談で申込決定]
   ↓
[管理者が Stripe 決済リンクを LINE/メールで送付]
   ↓
[受講生が Stripe 決済リンクで支払完了]
   ↓
[Stripe Webhook → 新サイトのバックエンド]
   ↓
[新サイトで受講生 ID 自動発行]
   ↓
[初回ログインリンクを受講生にメール送信]
   ↓
[受講生がログイン → パスワード設定]
   ↓
[マイページ・コンテンツアクセス開始]
```

### 銀行振込の扱い
- 銀行振込も Stripe で対応可能（ACH / Bank Transfer 機能）
- もしくは振込確認後、**管理者が新サイトで手動 ID 発行**

### 返金処理
- Stripe Dashboard で管理者が返金実行
- 新サイト側はステータスを「**返金処理中 → 返金済み**」と表示するだけ
