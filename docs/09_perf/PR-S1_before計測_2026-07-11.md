# PR-S1 速度改善・Before計測表（静的往復棚卸し） 2026-07-11

作成: Claude（きよむさん判定待ち）。指示書: `_Cursor指示_速度改善_応急_2026-07-11.md`。
本書は**コード無変更の静的解析**。S2以降はこの表との差分で往復数・直列段数を判定する。

## 前提・数え方
- 遅さの主犯 = 米国Vercel(iad1)⇄東京DB の距離 × 1画面で直列に連なる往復数（1往復 ~150–200ms）。
- 「1往復」= Supabaseへの1実行（`.from().select/insert`・`.rpc`・`storage.createSignedUrls`・`auth.getUser()`〔Authサーバへ検証する=ネットワーク往復〕・`auth.admin.listUsers`）。
- **クリティカル直列段数** = ページ処理を頭から見た「最長の直列チェーン」。`Promise.all`の束は1段（束の中は最深関数で数える）。TTFB ≈ 直列段数 × RTT + 計算。距離が固定なので**直列段数を減らすのが効く指標**。
- 推定TTFB(往復分) = 直列段数 × 175ms（US↔東京の代表RTT・目安。計算/描画は別途）。live実測は別枠（末尾）。

## 依存図（往復が長い上位3ページ）

### /meals（9段・全直列＝最短化の本命）
```
getUser(isBeta) → users ─┐
getUser(page) ───────────┤ ここまで認証
meal_logs → meal_log_items → createSignedUrls   （親子＋署名＝依存あり・直列は妥当）
daily_feedbacks        ┐
goal_sheets            ├─ 互いに独立なのに逐次awaitで直列（←束ねる対象）
getDailyCondition      │
shouldAskYesterday     │
getActiveFoods(food_table=マスタ) ┘
```
→ 独立4本(+マスタ)を `Promise.all` に束ねれば 9段→4段前後。

### /admin/daily（listUsersWithAlerts 二重）
```
requireAdmin(getUser→admin_users)
getDailyQueue → Promise.all[ listUsersWithAlerts() ★1(全件scan+auth.listUsers1000) , daily_feedbacks ]
getDailyDetail → users → meal→items→署名(3) → workout→items→menu(3)
                → Promise.all[ body, goal, carte, listUsersWithAlerts() ★2(全く同じ全件scan) , fb, lesson ]
                → lessons(title) → lessons(count) → daily_conditions
```
→ ★1の結果を★2に渡す（1回に）＝最重量の全件scan+listUsersを約7往復ぶん削減。lessons(count)/daily_conditions は束に入れられる。

### /（ホーム・beta）
```
getUser(page) → Promise.all(関所4) → Promise.all(12関数：各々が内部でgetUser+クエリ)
   最深: getMyLastWatched(3) / getMyUnreadCount(3) / getTodayActivity(getUser→6並列)
→ (beta) Promise.all(getRecordStreak[2], getTodayWorkout[4-5])  ←ここが後段に直列追加
   getTodayWorkout: getUser→getMyCurrentMenu(getUser+menu)→getMyProgress(getUser+progress)→logs→items
```
→ getUser集約（1回引いて各関数に渡す）＋ getMyProgress使い回し＋ beta後段をPromise.all(12)へ合流。

## S2での往復削減・当たり（この表からの差分目標）
